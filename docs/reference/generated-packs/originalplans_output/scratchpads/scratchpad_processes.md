# Scratchpad – Current Processes and Data Sources

This scratchpad collects notes about the existing workflow at Collision Engineers, gleaned from the provided files, GitHub repositories and job sheets.  It summarises the processes, datasets and requirements that will inform the planning documents.

## Intake Process

* **Email instructions** – Three Outlook inboxes receive new work instructions.  Human admin staff monitor the inboxes and record each job in a spreadsheet.  The spreadsheet has two sections: a **top section** for cases that are missing critical items and a **bottom section** for cases ready to be entered into EVA【500604108409406†screenshot】【531804138407202†screenshot】.  The “Missing” column indicates the reason a case can’t be processed (e.g., no images, no instruction, estimate missing)【909490197622550†screenshot】【525641506428545†screenshot】.

* **Images** – Many instruction emails arrive without photos of the damaged vehicle.  Images can be received in several ways: attached to the email, embedded in a PDF, or sent later via WhatsApp/email; sometimes they are uploaded to a third‑party system like Audatex.  If no images are available, staff must chase garages by WhatsApp or email to obtain them.  The **Principals** sheet in the job workbook lists providers and indicates where images usually come from (e.g., Accident Specialist group, customer WhatsApp)【188662199067677†screenshot】【173726810529521†screenshot】.

* **EVA data entry** – Once instructions and images are available, staff create a new case in EVA.  EVA requires fields such as **work provider (principal)**, **vehicle registration (VRM)**, **client claim reference**, **inspection address**, **mileage**, **valuation**, and an **internal case/po number**.  The job sheet shows that the internal case/po number is generated by taking the provider code (principal) and incrementing a counter each year.  Some fields (mileage, valuation) often need to be estimated or obtained from other services.

* **Valuations and Experian checks** – If an instruction doesn’t specify vehicle value, staff run a valuation service (e.g., Glass’s, or an AI tool) and download a companion report.  They also run an Experian check to identify adverse history.  These tasks currently require manual steps.

* **Box storage** – After entering a case in EVA, staff create a folder in Box named after the case/po number.  They upload the original email (.eml), instruction document, images, companion valuation report and any notes.  The **Box integration plan** for the existing document mapper outlines how to resolve target folders, pre‑flight uploads and avoid duplicates【644536497277043†L14-L31】.

* **Engineer dispatch** – Once a case is ready, an engineer is assigned.  The engineer performs the inspection and returns a report.  The EVA field `Type` is blank until an engineer has completed the assessment.

## Document Mapper and Providers

* The existing **CE Document Mapper** is a Python/Tkinter app that ingests documents and maps data to structured fields.  A transcript of the previous developer’s conversation with Claude explains that the app currently supports 13 fields in a specific order: **Work Provider, VRM, Vehicle Model, Claimant Name, Reference, Incident Date, Instruction Date, Inspection Date, Inspection Address, Accident Circumstances, VAT Status, Mileage, Mileage Unit**【399092012268091†L47-L62】.  The JSON export must preserve this order【399092012268091†L63-L64】.

* The `providers.json` file (in `docs/Settings Backup`) defines **provider presets** used by the mapper.  Each preset contains phrases used to detect a provider and mapping rules for each field.  For example, the **ALISON** provider uses a **two_labels** rule for the VRM (“Vehicle Reg: || Accident Date:”) and a **manual input** rule for the work provider (“ALISON”)【115463040094822†L3-L16】.  Presets also indicate whether to use the current date for inspection date and whether the provider is an engineer report.  These rules inform how the mapper extracts data from documents.

* The **Final Format Example 02.json** file demonstrates the expected JSON output from the mapper.  It includes fields like Work Provider, VRM, Vehicle Model, Claimant Name, Reference, Incident Date, Instruction Date, Inspection Date, Inspection Address, Accident Circumstances, VAT Status, Mileage and Mileage Unit.  In the example, **Work Provider** is “SBL” and **VRM** is “RJ62RTU”【230218250735069†L0-L14】.

* The repository also includes many example PDFs and DOC/DOCX instructions under `docs/Instructions`, as well as a backup of app settings.  These files provide test cases for extraction but are not individually analysed here.

## EVA and Sentry API

* EVA is the case management platform used by Collision Engineers.  The **Sentry API** guide describes how to integrate with EVA.  Key endpoints include:
  * **Instruct Claim** – creates a new claim in EVA.  It requires fields like `RequestFrom`, `ExternalRef`, `VehReg`, `ClmNo` (client claim number), `InspType` and others【859874972426525†L180-L232】.
  * **Claim Location Update** – updates the location on an existing claim, including address fields like `LocationName`, `Address` and `LocationType`【859874972426525†L275-L307】.
  * **Authority Status Update** and **Submit Note** – update status and add notes/comments to a claim【859874972426525†L343-L388】【859874972426525†L402-L438】.
  * **Submit Report** – sends a completed report, including valuation fields (vehicle value, salvage value), damage descriptions, parts lists and images【859874972426525†L517-L576】.
* The guide emphasises using a JWT token that expires after 5 minutes and should be refreshed frequently.

## Outlook and Box Integration Plans

* The **Automation Mode Plan** for the document mapper lists settings such as enabling automation, poll interval, automatically parsing new PDFs, exporting JSON automatically, uploading to Box automatically and marking an email as parsed only after the steps succeed【722788243396246†L2-L28】.

* The **Outlook Parsed Category Plan** states that emails should only be marked as `Parsed` after all steps (parse, review, JSON export and Box upload) succeed, and the integration should not move or delete messages; it should only update categories【353822838748330†L12-L22】.

* The **Box integration plan** explains that the app should resolve the target folder name (case number), check for duplicate files using Box’s search or metadata queries, pre‑flight the upload to avoid name conflicts, upload the original PDF and JSON export, apply metadata and remove the file from local storage【644536497277043†L14-L31】.

## Additional Data and Resources

* The developer created an **MCP server** to query DVSA/DVLA APIs for vehicle details such as MOT history.  This may assist with estimating mileage when it is missing.

* **Bulk MOT datasets** have been downloaded but their use cases need exploration.

* The **collisionautomation** repository provides a comprehensive product spec for a future case‑intake system, including features like a dashboard, evidence matching, engineer pack generation and AI processing.  These documents will inform the “New System” plans.

This scratchpad does not present final recommendations; it records the facts and context that underpin the forthcoming plans.

## Purpose

This file is the exhaustive planning ledger for CCC ideas found in reference material. It is intentionally broader than the Operational Core backlog. Items that are too large, speculative, dependency-heavy, or governance-gated are still planned here so they can be reviewed, phased, and promoted later.

This file does not authorize implementation by itself. Before build work starts, an item must be promoted into an owning plan or ticket with acceptance criteria, dependencies, verification, and governance gates.

## Audit Coverage

- `docs/reference/`: 549 on-disk files covering the reference index, original planning index, source manifest, raw evidence, normalized companions, reference data, Jam/FigJam exports, and test context material.
- `docs/reference/originalplanning/`: 143 on-disk files, including 138 Markdown files, 1 JSON manifest, and 4 ZIP archives. ZIP contents reviewed as planning evidence.
- `docs/reference/test-context/testprojectcontext/`: 43 files across both context packs.
- `docs/plans/operational-core/source_synthesis.md`: previous promotion and deferral map reviewed and converted into this all-ideas ledger, bringing the unique target inventory to 550 files.
- Binary raw files were reviewed through manifests, normalized companions, extracted data, provider matrices, and available structured parsers where practical.

## Promotion Rules

1. No implementation starts directly from this ledger.
2. Promotion requires an owning plan or ticket with status, owner, created date, last reviewed date, source links, roadmap milestone, dependencies, expected outputs, acceptance criteria, verification, archive target, and supersedes/superseded-by fields.
3. Long-horizon items remain planned in this ledger until promoted, superseded, or explicitly rejected by a later decision.
4. Governance-gated items require privacy, security, vendor, licensing, retention, and audit review before build work.
5. External partner, portal, payment, WhatsApp send, cloud OCR/document intelligence, valuation-provider, and EVA/Sentry submit work must not bypass human approval boundaries.
